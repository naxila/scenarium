Interpolation System

The new interpolation system provides a unified and intuitive way to access variables in scenarios, actions, and functions. It works similarly to variable scoping in programming languages.

Data Sources (Priority Order)

1. Local Context (Highest Priority)

Variables created by actions and functions in their local scope.

{{tempValue}} - Direct access
{{local.tempValue}} - Explicit prefix

2. Function/Action Parameters

Parameters passed to functions and actions.

{{param1}} - Direct access
{{params.param1}} - Explicit prefix

3. Scenario Data

Variables defined in the scenario's data block.

{{apiUrl}} - Direct access
{{data.apiUrl}} - Explicit prefix

4. Environment (Lowest Priority)

System variables automatically provided by the interpolation system.

{{timestamp}} - Direct access
{{env.timestamp}} - Explicit prefix

Available system variables:

  • {{timestamp}} or {{env.timestamp}} - Current date and time in ISO format
  • {{version}} or {{env.version}} - System version (0.1.0-alpha)
  • {{date}} or {{env.date}} - Current date in YYYY-MM-DD format
  • {{time}} or {{env.time}} - Current time in HH:MM:SS format

Note: These variables are generated automatically and are always available in any interpolation context.

Scope Management

The local context works like function scopes in programming languages:

  • Nested Scopes: Child functions can access parent variables
  • Variable Shadowing: Local variables can override parent variables
  • Automatic Cleanup: Scopes are cleared when functions/actions complete
  • Isolation: Each function/action manages only its own scope

Usage Examples

Basic Variable Access

// Scenario data
{
  "data": {
    "userName": "John",
    "apiUrl": "https://api.example.com"
  }
}

// In action
"text": "Hello {{userName}}! API: {{apiUrl}}"
// Result: "Hello John! API: https://api.example.com"

Local Context in Actions

// Action automatically creates local scope
// Variables are available for interpolation

// In action text
"text": "Message {{messageId}} sent: {{sent}}"
// Result: "Message 123 sent: true"

Nested Function Scopes

// Functions automatically inherit parent scope
// Child functions can access parent variables
// Local variables can override parent variables
// Scopes are automatically cleaned up when functions complete

// Example: Function A sets x=10, Function B can access x
// Function B sets y=20, Function C can access both x and y
// When functions complete, their local variables are cleared

Explicit Prefixes

// Force specific data source
"text": "Local: {{local.temp}}, Data: {{data.userName}}, Env: {{env.timestamp}}"

// Nested object access
"apiUrl": "{{data.api.baseUrl}}/{{data.api.version}}/users"

Best Practices

  • Use Direct Access: Prefer {{variableName}} over {{local.variableName}} when possible
  • Explicit Prefixes: Use prefixes when you need to be specific about data source
  • Scope Management: Scopes are automatically managed by the system
  • Variable Naming: Use descriptive names to avoid conflicts
  • Nested Objects: Use dot notation for nested object access
